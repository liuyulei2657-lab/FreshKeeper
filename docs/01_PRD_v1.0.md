## 1. 产品背景

### 1.1 市场现状

独居人群正在快速增长。根据统计数据，中国一人户家庭占比已超过 25%，独居生活成为不可忽视的社会趋势。独居用户自己采购食材、自己做饭，却普遍面临一个尴尬的问题：

**食材买回来，放进冰箱，然后就忘了。**

等到想起来的时候，蔬菜已经烂了，牛奶已经酸了，肉已经冻了半年。

### 1.2 用户痛点

- 冰箱里食材太多，记不住都有什么
- 记得买了某样东西，但想不起来什么时候买的、什么时候过期
- 每天打开冰箱不知道今天吃什么
- 经常在菜市场重复购买家里已有的食材
- 食物过期浪费严重，尤其绿叶蔬菜、鲜奶、豆制品

### 1.3 现有方案的问题

市面上现有方案要么太重（完整的库存管理系统，需要扫码、拍照、分类），要么太泛（备忘录、清单 App），要么不解决核心问题（菜谱 App 不关心你冰箱里有什么）。

没有一款产品能做到 **"3 秒打开，告诉今天吃什么"**。

### 1.4 为什么我们做

FreshKeeper 不做大而全的食品管理。我们只做一件事：**提醒你冰箱里的食物什么时候过期**。

## 2. 产品目标

### 2.1 核心目标

帮助独居用户减少食材浪费，降低决策成本。

### 2.2 用户目标

- 每天花最少的时间知道该吃什么
- 减少食材被遗忘而浪费
- 不需要学习就能用

### 2.3 业务目标

| 指标 | 目标值（上线 3 个月） |
|------|----------------------|
| 日活用户打开率 | > 60% |
| 食材录入后消耗率 | > 70% |
| 次日留存 | > 40% |
| 用户好评率 | > 90% |

### 3.1 典型用户：小林

| 属性 | 内容 |
|------|------|
| 年龄 | 26 岁 |
| 职业 | 互联网运营 |

| 采购习惯 | 周末一次性采购，偶尔下班补货 |

**周一早上：** 小林赶着上班，打开冰箱拿牛奶，发现上周买的生菜已经烂了。她叹了口气扔掉，心想"下次一定记得先吃它"。

**周四晚上：** 小林下班路过菜市场，买了把菠菜，回家发现冰箱里还有一把上周买的菠菜——已经蔫了。

- "我就想知道冰箱里有什么快过期了"
- "别让我填太多东西，我懒"

- "离线也能用，我地铁里经常没信号"

## 5. MVP

### 5.1 MVP 范围

MVP 只包含核心闭环：**录入 → 查看 → 提醒 → 消耗**。

所有超出这个闭环的功能都不进 MVP。

### 5.2 MVP 功能清单

| 模块 | 功能 | 说明 |
|------|------|------|
| 录入 | 快速添加食材 | 选择食材名称 + 过期日期，10 秒完成 |
| 录入 | 常用食材快捷选择 | 预置常见食材列表，一键选择 |
| 查看 | 首页食材列表 | 按过期日期排序，快过期靠前 |
| 查看 | 今天吃什么 | 顶部展示即将过期的食材 |
| 管理 | 标记已吃掉 | 一键标记消耗 |
| 管理 | 标记已丢弃 | 一键丢弃 |
| 通知 | 本地通知 | 每日定时检查并发送过期提醒 |
| 设置 | 提醒时间设定 | 用户可选通知时间 |
| 设置 | 提前天数设定 | 用户可选提前 1/2/3 天提醒 |

### 5.3 MVP 不包含

- 食材照片拍照识别
- 条形码扫描
- 购物清单
- 数据导出
- 分类管理（MVP 使用固定分类）
- 多语言
- 主题切换
- 统计报表

## 11. 通知逻辑

### 11.1 设计原则

- **全部本地通知**，不依赖任何推送服务
- **静默无打扰**，只在食材过期或即将过期时通知
- **通知即行动**，通知内容让用户能直接做决策

### 11.2 触发时机

检查时机：App 启动时 + 每日定时检查。

定时检查机制：使用系统 AlarmManager（Android）/ UNUserNotificationCenter（iOS）调度每日检查任务。

### 11.3 通知规则

| 条件 | 通知内容 | 示例 |
|------|----------|------|
| 今天过期食材 >= 1 | 发送"今日过期"通知 | "🥬 牛奶、菠菜今天过期，记得吃掉" |
| 即将过期食材 >= 1 | 发送"即将过期"通知 | "🥬 你有 3 样食材即将过期" |
| 今天过期 = 0，即将过期 = 0 | 不发送通知 | — |

### 11.4 通知去重

- 单日不重复发送同类型通知
- 如果用户已打开 App 查看过，当天不再发送

### 11.5 设置项

| 设置 | 默认值 | 说明 |
|------|--------|------|
| 提醒时间 | 08:00 | 每天检查并通知的时间 |
| 提前提醒天数 | 2 天 | 过期前 N 天开始提醒 |
| 启用通知 | 开启 | 总开关 |

## 12. 状态流转

### 12.1 食材状态定义

| 状态 | 定义 | 显示 |
|------|------|------|
| normal | 未过期，且距离过期 > 3 天 | 灰色"还有 N 天" |
| expiring | 即将过期，距离过期 <= 3 天 | 黄色"还有 N 天" |
| expired | 已过期 | 红色"已过期 N 天" |
| consumed | 已吃掉 | 不显示在首页 |
| discarded | 已扔掉 | 不显示在首页 |

```
        │         └────┬─────┘         │
        │              │ 时间流逝       │
        │              ▼               │
        │         ┌──────────┐         │

        │         └────┬─────┘
        │              │
        │     ┌────────┼────────┐
        │     │        │        │
        ▼     ▼        ▼        ▼
   ┌────────┐ ┌──────────┐ ┌────────┐
   │consumed│ │ discarded│ │ 再次   │
   └────────┘ └──────────┘ │录入    │
                            └────────┘
```

### 12.3 状态自动流转

- 食材录入后状态为 `normal`
- 每天零点自动检查：`normal` → 距离过期 <= 3 天 → `expiring`
- 每天零点自动检查：`expiring` → 已过当天零点 → `expired`

### 12.4 用户操作

| 操作 | 原状态 | 目标状态 |
|------|--------|----------|
| 标记"已经吃掉" | 任意非终态 | consumed |
| 标记"已经扔掉" | 任意非终态 | discarded |
| 修改过期日期 | normal / expiring / expired | 重新计算 |
| 删除食材 | 任意状态 | 从数据库物理删除 |
